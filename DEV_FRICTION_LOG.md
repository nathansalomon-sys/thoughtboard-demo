## Friction Points

### [2026-03-08 23:30] @cloudflare/workers-types superseded but still the default manual setup pattern
- **What happened:** Scaffolded project manually using `@cloudflare/workers-types` in `package.json` and `tsconfig.json` — the long-standing standard pattern. Running `wrangler types` immediately flagged it as deprecated and told us to uninstall it in favour of the generated runtime types (`worker-configuration.d.ts`).
- **Impact:** Required an extra uninstall + tsconfig edit immediately after setup. Any developer following older blog posts or docs will hit this same correction.
- **Resolution:** Removed `@cloudflare/workers-types` from `package.json` and `tsconfig.json`; rely entirely on `worker-configuration.d.ts` generated by `wrangler types`.
- **Source:** `wrangler types` CLI output / [Cloudflare Workers TypeScript docs](https://developers.cloudflare.com/workers/languages/typescript/#generate-types)

### [2026-03-08 23:55] `wrangler workflows trigger` params syntax undocumented / flag name misleading
- **What happened:** Attempted `wrangler workflows trigger feedback-pipeline --payload '{}'` based on common CLI conventions. Wrangler returned an error: "Unknown argument: payload". The actual syntax is a positional argument: `wrangler workflows trigger <name> <params>`.
- **Impact:** Failed trigger attempt; required reading the help output to find correct syntax. ~5 minutes lost.
- **Resolution:** Used `wrangler workflows trigger feedback-pipeline '{}'` (params as second positional, no flag).
- **Source:** `wrangler workflows trigger --help` output

### [2026-03-09 00:05] Workflows "Too many subrequests" with Promise.all fan-out
- **What happened:** Designed pipeline with `Promise.all(batch.map(row => step.do(...)))` — 20 concurrent steps, each making 1 Workers AI call + 1 D1 INSERT. The workflow errored with `"Too many subrequests"` after the first fetch step succeeded.
- **Impact:** Full pipeline redesign required. Workers subrequest limits apply per Worker invocation; concurrent `step.do()` callbacks sharing the orchestrator context multiply the subrequest count. ~30 minutes lost diagnosing and redesigning.
- **Resolution:** Replaced per-item steps with per-batch steps (20 items each). Inside each step, items are classified sequentially and all D1 inserts are done in a single `DB.batch()` call, keeping subrequests per step to ~21 (20 AI + 1 D1 batch).
- **Source:** Trial and error / Cloudflare Workers subrequest limit docs

### [2026-03-09 00:15] Workers AI `response_format: { type: "json_object" }` rejected — model expects `json_schema`
- **What happened:** Used `response_format: { type: "json_object" }` with `@cf/meta/llama-3.1-8b-instruct`, matching the OpenAI API format. Workers AI rejected it with `AiError: 9015: unknown variant 'json_object', expected 'json_schema'`.
- **Impact:** Pipeline failed on every retry attempt. The Workers AI model has a different response_format API than OpenAI — `json_object` is not valid; the supported type is `json_schema` (with a schema object). ~15 minutes lost on diagnosis + rewrite.
- **Resolution:** Removed `response_format` entirely. Instead, the model is instructed via the system prompt to return only JSON, and the response is parsed by extracting the first `{...}` match via regex — making it robust to any surrounding text.
- **Source:** `wrangler workflows instances describe` error output

### [2026-03-09 00:25] Workflows subrequest counter appears to accumulate across steps, not reset per step
- **What happened:** After fixing `response_format`, redesigned pipeline to use 30 sequential batch steps (20 items/step, 21 subrequests each). Batches 0 and 20 (items 0–39) succeeded, but batch 40 (items 40–59) consistently failed with "Too many subrequests". Per-step budget should be 50+, yet the 3rd batch fails — consistent with subrequests accumulating across the workflow instance rather than resetting per step invocation.
- **Impact:** Required a second full architecture redesign. The per-batch-step model that's idiomatic for Workflows is unusable when AI + D1 calls are made inside each step. Final solution collapses all AI processing into a single large step.
- **Resolution:** Replaced 30 batch steps with 2 steps total: fetch-unprocessed + process-all. All 600 AI calls run sequentially inside "process-all" (timeout: 30 min), and results are written with a single `DB.batch()` at the end.
- **Source:** Trial and error; behavior not documented in Cloudflare Workflows limits docs

### [2026-03-09 01:00] Workflows subrequest budget is per-instance, not per-step — makes AI-heavy pipelines impractical
- **What happened:** Even after collapsing all processing into a single `step.do()`, the step failed at ~30 AI calls with "Too many subrequests". Subsequent retries failed immediately (0-second duration), confirming the budget was already exhausted from the first attempt. The Cloudflare Workers subrequest limit (50 free / 1000 paid) is tracked at the Workflow instance level, not per-step invocation. This makes Workflows unsuitable as the sole orchestrator for pipelines that require many external service calls (AI, D1) in one run.
- **Impact:** Workflows architecture works for incremental processing (few items at a time) but cannot process 600 items in one instance. Required switching initial bulk processing to HTTP batch endpoint calls — each HTTP call is a fresh Worker invocation with an independent subrequest budget.
- **Resolution:** Added `/pipeline/run-batch` POST endpoint to the Worker. A local `scripts/run-pipeline.js` fetches unprocessed IDs via wrangler CLI and drives the endpoint in sequential batches of 20, each within budget. The `FeedbackPipeline` Workflow class remains for incremental future runs.
- **Source:** Trial and error / observed behavior across 3 pipeline architecture iterations

### [2026-03-09 02:30] Local D1 requires manual migration + pipeline run — wrangler dev starts empty
- **What happened:** `wrangler dev` starts with a completely empty local D1 instance. Running `wrangler d1 migrations apply --local` is required to create tables; then the full classification pipeline must be run against localhost to populate `feedback_processed`. Without this, every API route returns a 500 with `D1_ERROR: no such table: feedback_processed`.
- **Impact:** Dashboard showed blank/500 errors on first local run. Two separate steps needed that aren't obvious from wrangler docs.
- **Resolution:** `npx wrangler d1 migrations apply thoughtboard-db --local`, then drive `/pipeline/run-batch` 30× via curl loop. Also note: `scripts/run-pipeline.js` hardcodes the remote URL — `WORKER_URL` env var is NOT read from `process.env`, so the script always hits the deployed worker, not localhost.
- **Source:** Trial and error

### [2026-03-09 02:00] Chart.js annotation plugin not bundled in the UMD build
- **What happened:** Designed the sentiment trend chart to use `chartjs-plugin-annotation` for a y=0 reference line (neutral baseline). The UMD CDN build of Chart.js 4 does not include this plugin — the `annotation` options block is silently ignored unless the plugin is registered separately.
- **Impact:** Reference line does not render without adding a second CDN script. Chose to remove the annotation block from options rather than add another CDN dependency for a cosmetic feature.
- **Resolution:** Removed the `annotation` plugin options. The y=0 neutral baseline is implied by the scale min/max (-1 to 1); the gradient fill already visually distinguishes negative from positive territory.
- **Source:** Chart.js 4 docs / CDN bundle contents
